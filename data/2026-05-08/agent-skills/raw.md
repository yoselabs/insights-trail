### Batch 7 — Official Docs & Technical Specs

From https://code.claude.com/docs/en/plugins (full fetch):
- Plugin structure: `.claude-plugin/plugin.json` manifest + `skills/`, `agents/`, `hooks/`, `.mcp.json`, `.lsp.json`, `monitors/`, `bin/`, `settings.json`
- Skills = namespaced `/plugin-name:skill-name`; standalone = short `/skill-name`
- `--plugin-dir` flag for local dev; `--plugin-url` flag for URL-distributed .zip
- `/plugin install {name}@marketplace-repo`
- Skill arguments: `$ARGUMENTS` placeholder
- Progressive disclosure: Claude auto-invokes skills by semantic matching
- LSP server support in plugins (real-time code intelligence)
- Background monitors: `monitors/monitors.json` watches logs/files
- `settings.json` in plugin can set default agent
- Submit to official marketplace: claude.ai/settings/plugins/submit or platform.claude.com/plugins/submit
- Plugin hints: recommend installation from CLI
- Version management: explicit `version` field or git commit SHA

From https://www.morphllm.com/claude-code-skills-mcp-plugins (full fetch):

## FUSE/RERANK/CLUSTER

### Cluster 1: The Claude Code Extension Taxonomy
High relevance; official docs + practitioner guides + HN debate all converge on this architecture question.

Core taxonomy (from official docs + morphllm guide):
- **Skill**: 30-50 token overhead; procedural instructions in SKILL.md; model-invoked via semantic match; context-efficient via progressive disclosure
- **Plugin**: bundled package (skills + agents + hooks + MCP + LSP + monitors); shareable; namespaced `/plugin:skill`; distributed via marketplace catalog
- **MCP Server**: external tool connectivity; 1-50k tokens upfront; client-agnostic protocol; use for real-time data/API/browser
- **Hook**: event handlers (PostToolUse etc.); minimal overhead; automation
- **Slash Command/Agent**: orchestration primitives
